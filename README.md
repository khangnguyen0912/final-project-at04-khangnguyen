# Final Project AT04 - API/UI Automation

Automation test project for `https://book.anhtester.com/` using API and UI flows.

## Tech Stack

- Python + pytest
- Playwright (sync + async usage)
- allure-pytest
- openpyxl
- requests
- python-dotenv

## Project Structure

- `tests/ui`: UI test suite
- `tests/api`: API test suite
- `tests/api_ui`: mixed API-UI tests
- `tests/conftest.py`: shared fixtures
- `tests/pytest_hooks.py`: pytest hooks (`pytest_runtest_makereport`)
- `utils/support/account_recovery.py`: account restore logic for UI test isolation
- `utils/support/ui_session.py`: Playwright storage state handling for UI session reuse
- `api/payload`: request payload builders
- `data/test_cases/update_profile.xlsx`: test data source
- `screenshots`: auto-captured screenshots on failed UI tests
- `storage/auth.json`: persisted Playwright storage state
- `logs/test_run_*.log`: execution logs per test run

## Test Data Conventions

`data/test_cases/update_profile.xlsx` is the main source for data-driven test cases.

Supported value conventions in loaders:

- `EMPTY` -> empty string `""`
- `NULL` -> Python `None`
- `TRUE` / `FALSE` -> boolean mapping for expected result flags
- `{{random_generated_email}}` -> generated by `utils.randoms.random_email()`
- `{{random_new_password}}` -> generated by `utils.randoms.random_password()`
- `{{random_confirmed_password}}` -> auto-sync to generated/new password

## Run Tests

Run all tests:

```bash
pytest
```

Run UI suite:

```bash
pytest tests/ui
```

Run API suite:

```bash
pytest tests/api
```

Run one test file:

```bash
pytest tests/ui/test_update_profile.py
pytest tests/api/api_patch_profile/single_field/test_update_email_field.py
```

Run one case by keyword:

```bash
pytest tests/ui/test_update_profile.py -k "reload"
```

## Reports and Artifacts

Generate Allure results:

```bash
pytest tests/ui --alluredir=allure-results
```

Open Allure report (requires Allure CLI):

```bash
allure serve allure-results
```

Failed UI test screenshots are saved to:

- `screenshots/<normalized_case_name>_<monDD_YYYY_HHMMSS>.png`

Example:

- `verify_that_user_can_access_my_profile_page_mar05_2026_140549.png`

Execution logs are saved to:

- `logs/test_run_<YYYYMMDD_HHMMSS>.log`

UI session storage state is reused from:

- `storage/auth.json`

## Account Recovery Behavior (UI Tests)

To keep UI tests isolated, the framework restores the default account (`EMAIL/PASSWORD` in `.env`) before and after each UI test case.

Flow summary:

- Try login using `.env` credentials.
- If login response message is `Invalid password.`, attempt to reset password back to `.env` password via `PATCH /api/profile`.
- If login response message is `User not found.`, register the account again using payload from `api/payload/register_payload.py`.

The registration payload builder is:

- `api.payload.register_payload.build_register_payload(email, password)`
